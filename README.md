# CodeIU

CodeIU is a full-stack competitive programming platform built as a monorepo. It combines a Next.js frontend with a Node.js microservices backend for authentication, problem management, submissions, contests, and notification delivery.

This README is the canonical entry point for the repository. It summarizes what is actually present in the codebase today, how the pieces fit together, and what you need to run or extend the system locally.

## Repository at a glance

- `CodeIU-Frontend/`: Next.js 15 application, App Router, Zustand stores, Monaco-based coding workspace, contest UI, admin flows
- `CodeIU-Backend/`: five backend services plus Docker, verification scripts, ECS task definitions
- `scripts/`: root helper scripts for starting all services and clearing ports
- `logs/`: output generated by the root dev launcher
- `api/postman/`: Postman collections for the backend services
- top-level `*.md`: product notes, implementation plans, and architecture references

## Architecture

```text
Browser
  |
  v
Next.js frontend (CodeIU-Frontend)
  |
  +--> /api/v1/auth/*         -> AuthService
  +--> /api/v1/admin/*        -> AuthService admin routes
  +--> /api/v1/problem/*      -> ProblemService
  +--> /api/v1/playlist/*     -> ProblemService playlist routes
  +--> /api/v1/submission/*   -> SubmissionService history/stats
  +--> /api/v1/execute/*      -> SubmissionService run/submit routes
  +--> /api/v1/contest/*      -> ContestService
  +--> /api/v1/notification/* -> NotificationService
  |
  v
Backend services
  |
  +--> PostgreSQL via Prisma
  +--> Redis for queues, cache, socket fan-out, sessions
  +--> Judge0 for code execution
  +--> Socket.IO for real-time updates
  +--> Nodemailer for transactional email
```

### Core backend services

| Service | Port | Main responsibility | Route prefix |
| --- | --- | --- | --- |
| AuthService | `8020` | login, signup, profile, refresh tokens, Google OAuth, admin user management | `/auth/api/v1` |
| ProblemService | `8000` | problem CRUD, search, solved tracking, playlists, resources | `/problem/api/v1` |
| SubmissionService | `8080` | run code, submit code, Judge0 integration, submission history, profile stats | `/submission/api/v1` |
| ContestService | `8090` | contest CRUD, registration, frozen bundles, live leaderboard, contest workers | `/contest/api/v1` |
| NotificationService | `8030` | BullMQ worker for email jobs plus health endpoint | `/health` and worker queue |

### Shared infrastructure

- Redis is used by sessions, queues, caching, contest updates, and real-time submission delivery.
- Prisma is used in AuthService, ProblemService, SubmissionService, and ContestService.
- Judge0 is required for code execution in SubmissionService and is also referenced by ProblemService utilities.
- Socket.IO is used by SubmissionService and ContestService for real-time judge and leaderboard events.

## API routing model

The frontend does not call each backend directly in normal app code. It uses Next.js rewrites from `CodeIU-Frontend/next.config.mjs`.

### Local development behavior

If `NEXT_PUBLIC_DIRECT_ALB_URL` is unset or contains `localhost`, frontend requests are rewritten to local service ports:

- `/api/v1/auth/*` -> `http://localhost:8020/auth/api/v1/auth/*`
- `/api/v1/admin/*` -> `http://localhost:8020/auth/api/v1/admin/*`
- `/api/v1/submission/*` -> `http://localhost:8080/submission/api/v1/submission/*`
- `/api/v1/execute/*` -> `http://localhost:8080/submission/api/v1/execute/*`
- `/api/v1/contest/*` -> `http://localhost:8090/contest/api/v1/contest/*`
- `/api/v1/notification/*` -> `http://localhost:8030/notification/api/v1/notification/*`
- `/api/v1/playlist/*` -> `http://localhost:8000/problem/api/v1/playlist/*`
- `/api/v1/:service/*` -> `http://localhost:8000/:service/api/v1/:service/*`

### Production behavior

If `NEXT_PUBLIC_DIRECT_ALB_URL` is set to a non-local value, those same paths are proxied through the external ALB base URL instead.

## Environment and configuration

### Important reality checks

- Backend services load `.env.development` in local development and `.env.production` in production. Several service-specific READMEs still mention `.env`, but the code loads the environment-specific filenames.
- `CodeIU-Backend/docker-compose.yml` expects `CodeIU-Backend/envs/*.env` files, but that `envs/` directory is not committed in this repository.
- The current frontend uses `NEXT_PUBLIC_DIRECT_ALB_URL`; the older `NEXT_PUBLIC_API_URL` shown in some docs is not the main setting used by the current code.
- Next.js middleware verifies auth cookies locally and requires `JWT_SECRET` to be available to the frontend runtime for protected/admin route handling.

### Frontend environment

Create `CodeIU-Frontend/.env.local`.

| Variable | Required | Purpose |
| --- | --- | --- |
| `JWT_SECRET` | Yes | used by `src/middleware.js` to validate auth cookie payloads |
| `NEXT_PUBLIC_DIRECT_ALB_URL` | No for local, yes for remote API mode | base URL for production-style proxying |
| `INTERNAL_GATEWAY_URL` | Optional | server-side internal base URL override used by `src/lib/urls.js` |
| `NEXT_PUBLIC_CLOUDINARY_CLOUD_NAME` | Optional | resource upload UI |
| `NEXT_PUBLIC_CLOUDINARY_UPLOAD_PRESET` | Optional | resource upload UI |

### Backend service environment

Create `.env.development` in each backend service directory unless you are using the ECS/Docker deployment path.

| Service | Core required variables | Notes |
| --- | --- | --- |
| AuthService | `PORT`, `DATABASE_URL`, `JWT_SECRET`, `SESSION_SECRET`, `FRONTEND_URL`, `REDIS_URL`, `CORS_ORIGINS` | optional `REFRESH_TOKEN_SECRET`, `CLIENT_ID`, `CLIENT_SECRET`, `REDIRECT_URL` for Google OAuth |
| ProblemService | `PORT`, `NEON_DB`, `JWT_SECRET`, `REDIS_URL`, `CORS_ORIGINS` | `NEON_DB` is the actual Prisma datasource variable name in this service |
| SubmissionService | `PORT`, `DATABASE_URL`, `JWT_SECRET`, `REDIS_URL`, `CORS_ORIGINS`, `JUDGE0_API_URL`, `PROBLEM_SERVICE_URL` | optional `JUDGE0_API_KEY` and host-specific Judge0 settings |
| ContestService | `PORT`, `DATABASE_URL`, `JWT_SECRET`, `REDIS_URL`, `CORS_ORIGINS` | usually also needs internal service URLs such as `AUTH_SVC`, `PROBLEM_SVC`, `SUBMISSION_SVC`, `NOTIFY_SVC` |
| NotificationService | `PORT`, `REDIS_URL`, `EMAIL_USER`, `EMAIL_PASS` | optional `EMAIL_FROM_NAME`, `SMTP_HOST`, `SMTP_PORT`, `CORS_ORIGINS` |

Optional production-oriented variables appear in the codebase as well, including Cloudflare purge settings, Redis host/port overrides, worker concurrency values, and service JWT settings.

## Local development

### Prerequisites

- Node.js 18+ and npm
- Redis running locally or reachable remotely
- PostgreSQL databases for the services that use Prisma
- a working Judge0 instance for run/submit flows
- valid environment files for every service you want to boot

### Install dependencies

There is no root workspace manager here. Install dependencies package by package:

```bash
cd CodeIU-Frontend && npm install
cd ../CodeIU-Backend/services/AuthService && npm install
cd ../ProblemService && npm install
cd ../SubmissionService && npm install
cd ../ContestService && npm install
cd ../NotificationService && npm install
```

If you are setting up Prisma-backed services for the first time, also run their generate and migration/bootstrap commands as required by each package.

### Fastest source-based dev workflow

From the repository root:

```bash
npm run dev
```

What this does:

- starts all five backend services from source
- starts the Next.js frontend
- writes service logs into `logs/`
- keeps running until interrupted

Generated log files:

- `logs/auth.log`
- `logs/problem.log`
- `logs/submission.log`
- `logs/contest.log`
- `logs/notification.log`
- `logs/frontend.log`

To follow logs:

```bash
tail -f logs/frontend.log
tail -f logs/submission.log
```

To stop or clean up ports:

```bash
npm run kill
```

Note: `scripts/kill-ports.sh` uses `sudo fuser` on ports `3000`, `8000`, `8020`, `8030`, `8080`, and `8090`.

### Manual service startup

If you prefer to boot services individually:

```bash
cd CodeIU-Backend/services/AuthService && npm run dev
cd CodeIU-Backend/services/ProblemService && npm run dev
cd CodeIU-Backend/services/SubmissionService && npm run dev
cd CodeIU-Backend/services/ContestService && npm run dev
cd CodeIU-Backend/services/NotificationService && npm run dev
cd CodeIU-Frontend && npm run dev
```

### Docker Compose path

The backend contains a `docker-compose.yml`, but it is important to understand what it is for:

- it pulls prebuilt service images from `ghcr.io`
- it starts Redis locally
- it expects environment files under `CodeIU-Backend/envs/`
- it does not build the local source tree
- it does not provision PostgreSQL or Judge0 for you

Use it if you already have the deployment-style env files and image access. Do not assume it is a zero-config local source development setup.

## Testing, linting, and verification

### Frontend

```bash
cd CodeIU-Frontend
npm run lint
npm run build
```

### Backend services

Each backend service exposes at least:

- `npm run lint`
- `npm run lint:fix`
- `npm run format`
- `npm run test`

Services with Prisma also include generate and migrate-related scripts.

### Bulk backend verification

From `CodeIU-Backend/`:

```bash
./verify-all.sh
```

This script iterates through all backend services and runs:

- `npm run lint:fix`
- `npm run format`
- `npm run test`

### Endpoint smoke check

From `CodeIU-Backend/` after the services are running:

```bash
./verify_endpoints.sh
```

It checks:

- `http://localhost:8020/auth/health`
- `http://localhost:8090/contest/api/v1/contest/contests`
- `http://localhost:8000/problem/api/v1/problem/getAllProblem`
- `http://localhost:8080/submission/api/v1/submission/getAllSubmission`
- `http://localhost:8030/health`

Expected behavior includes a `401` on the submission history route when no auth cookie is present.

## Operational notes

- Root `npm run clean` recursively removes every `node_modules` directory under the repository.
- `scripts/start-all.sh` assumes all dependencies are already installed and every service has valid configuration.
- `ContestService`, `SubmissionService`, and `ProblemService` each run API and worker processes together under their `dev` command.
- `NotificationService` is primarily a worker with a small Express health server attached.
- The repository contains ECS task definitions under `CodeIU-Backend/ecs-task-definitions/`, which shows the intended production deployment model.

## Known gaps and caveats

- The committed documentation under subdirectories is not fully synchronized with the current runtime behavior. This root README corrects the biggest mismatches, especially env filenames and frontend proxy settings.
- Not every roadmap item described in `README_FEATURES.md` is implemented in the repository. Treat that file as product planning, not as a guaranteed feature inventory.
- Some shell helpers are developer-machine specific, especially `scripts/start-all.sh`, which hardcodes the repository base path.

## Suggested starting points

If you are new to the repo:

1. Read this file.
2. Configure the frontend and one backend service first, usually `AuthService`.
3. Bring up Redis and confirm health endpoints.
4. Start the full stack with `npm run dev`.
5. Use the Postman collections or the frontend UI to validate auth, problem listing, and submission flow.
